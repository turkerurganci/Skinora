## Yapılan İşler

T11.1 retrospektifinde ortaya çıkan 5 disiplin ihlali patterninin (T13-T20 dönemi) bir daha tekrarlanmaması için dört savunma katmanı eklendi:

- **Katman A — Startup check (skill düzeyi):** `/task TXX` ve `/validate TXX` skill'leri Adım 0'da main CI son 3 run'ını kontrol eder; biri bile FAIL ise **HARD STOP**. Rasyonelizasyon ("lokal temiz", "ilgisiz", "sadece docker-publish") yasak.
- **Katman B — Pre-push CI guard (hook düzeyi):** `scripts/git-hooks/pre-push` push öncesinde mevcut branch'in son CI run'ını kontrol eder; failure/cancelled/timed_out ise push bloklanır. `gh` yoksa WARN + geçer (fail-open). Bypass → `BYPASS_LOG.md`'ye `[ci-failure]` kayıt.
- **Katman C — Validator finding kuralı:** `validate.md` Faz 1 Adım 7a'da task branch CI FAIL ise S2 Kırılma finding'i, sessizce geçilemez; `INSTRUCTIONS.md §3.3`'e paralel "validator CI rasyonelizasyon yasağı" maddesi eklendi.
- **Katman D — Task chat Bitiş Kapısı:** `task.md` sonunda 4 maddelik check liste (branch push + PR create + PR numarası rapora yazıldı + CI run başladı) — hepsi ✓ olmadan task "yapım bitti" sayılmaz; "PR: Henüz oluşturulmadı" → otomatik BLOCKED. Bundled PR yasağı eklendi.
- **BYPASS_LOG düzeltmesi:** T11.1'de hatalı retro olarak "direct-push" kaydedilen T14 satırı kaldırıldı (T14 aslında PR #8 ile merge olmuş, `gh pr view 8` ile doğrulandı). T15+T16 retro kayıtları birleştirilip `[bundled-pr]` pattern'iyle yeniden sınıflandırıldı. Log header'ına `[direct-push]`/`[ci-failure]`/`[bundled-pr]` önek konvansiyonu eklendi.
- **09 §21.5 eklendi:** "Notification & Guard Katmanları" bölümü — dört katman birlikte belgelendi (tek savunma katmanı yetmiyor).

## Known Limitations / Follow-up

0. **Bitiş Kapısı düzeltmesi (kullanıcı uyarısı, 2026-04-12):** İlk taslakta Bitiş Kapısı 4. maddesi "CI run başladı mı" olarak yazılmıştı — bu T13-T20 "kırık CI'yi validator'a havale" pattern'inin ta kendisi. Kullanıcı uyarısı üzerine task.md + 09 §21.5 güncellendi: artık CI run **tamamlandı + success** zorunlu, aksi halde task "yapım bitti" sayılmaz. 5 maddelik check list'e çıkarıldı. Concurrency ile cancel olan run'lar `failure` sayılmaz; en son tamamlanmış run'a bakılır notu eklendi.
1. **End-to-end CI-failure block smoke:** Gerçek failed CI run üzerinden push bloklama end-to-end testi yapılmadı; parse + dispatch + block kodu izole test edildi. Validator bu task'ın kendi PR CI run'ı üzerinden gerçek davranışı doğrulayabilir.
2. **Hook jq bağımlılığı:** Windows'ta jq yaygın değil, bu yüzden grep/cut parser seçildi. GitHub CLI JSON çıktı formatı değişirse parser kırılabilir — `gh run list --json conclusion,status` alanları stabil kabul edildi.
3. **`gh` auth downgrade:** `gh` var ama auth yoksa `skip` döner — bu fail-open davranışı direct-push kuralını etkilemez, sadece CI guard skipped.
4. **Startup check tek yönlü:** Skill'ler kullanıcıdan sebebini "sorar" ama kullanıcı cevabını doğrulamak skill'in elinde değil; bu disiplin kuralı (model cevabı itirazsız kabul etmez) olarak INSTRUCTIONS.md §3.3'te kayıtlı.
5. **Bundled PR tespiti otomatik değil:** Task.md "Bitiş Kapısı" bundled PR yasağını yazıyor ama mekanik olarak yakalamıyor — bu süreç kuralıdır, ihlal retrospektiften görünür (BYPASS_LOG `[bundled-pr]` pattern ile).
